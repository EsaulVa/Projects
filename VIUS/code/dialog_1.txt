## 2. Ошибка в `compute_grad_Phi` (теоретическая)

В `inverse_method_fixed.py` функция `compute_grad_Phi` вычисляет ковариантные компоненты градиента связи. Давайте проверим формулу.

В отчёте (формула 6):
$$g_\alpha = b_{\alpha\beta} \dot{u}^\beta$$

где $b_{\alpha\beta}$ — ковариантные компоненты II формы $(L, M, N)$, а $\dot{u}^\beta$ — **контравариантные** компоненты вектора нити.

В коде:
```python
tau_u = E * u_prime + F * v_prime   # ковариантные компоненты τ_α
tau_v = F * u_prime + G * v_prime
...
b_u_u = guu * L + guv * M           # b^u_u (смешанный)
...
dPhidu = -lam * (b_u_u * tau_u + b_u_v * tau_v)
```

Здесь `b_u_u` — смешанная компонента $b^u_u = g^{uu}L + g^{uv}M$, а `tau_u` — ковариантная $\tau_u = g_{u\beta}\dot{u}^\beta$. Их произведение даёт $b^u_\beta \tau_\beta$, что **не равно** $b_{u\beta}\dot{u}^\beta$ для общей поверхности.

**Для поверхности вращения** ($F=0$) это случайно компенсируется:
- $\tau_u = E \cdot u'$, $b^u_u = L/E$ → произведение $= L \cdot u'$
- $\tau_v = G \cdot v'$, $b^u_v = M/G$ → произведение $= M \cdot v'$

Так что для вращения формула работает, но это **нестабильное совпадение**. Если вы обобщите на другие поверхности — сломается.

**Исправление** — явно использовать ковариантные компоненты II формы:
```python
def compute_grad_Phi(surface, u, v, u_prime, v_prime, lam):
    L, M, N_ff = surface.second_fundamental_form(u, v)
    # g_α = b_{αβ} · τ^β  (ковариантные компоненты градиента)
    dPhidu = -lam * (L * u_prime + M * v_prime)
    dPhidv = -lam * (M * u_prime + N_ff * v_prime)
    return dPhidu, dPhidv
```

Или, если хотите через смешанный тензор:
```python
    dPhidu = -lam * (b_u_u * u_prime + b_u_v * v_prime)
    dPhidv = -lam * (b_v_u * u_prime + b_v_v * v_prime)
```

## 3. Проблема нормали и II формы для масштабированной поверхности

Проверим, правильно ли `ScaledPiecewisePolynomialRevolution` наследует геометрию.

Для масштабированной поверхности:
- $\mathbf{r}(u,v) = (k r(u)\cos v,\; k r(u)\sin v,\; u)$
- $\mathbf{r}_u = (k r'\cos v,\; k r'\sin v,\; 1)$
- $\mathbf{r}_v = (-k r\sin v,\; k r\cos v,\; 0)$

Нормаль (ненормированная):
$$\mathbf{n}_0 = (-k r\cos v,\; -k r\sin v,\; k^2 r r')$$

Норма: $\|\mathbf{n}_0\| = k r \sqrt{1 + k^2 r'^2}$

Нормированная:
$$\mathbf{n} = \frac{(-\cos v,\; -\sin v,\; k r')}{\sqrt{1 + k^2 r'^2}}$$

Для базовой ($k=1$): $\mathbf{n} = \frac{(-\cos v,\; -\sin v,\; r')}{\sqrt{1 + r'^2}}$.

**Вывод:** нормали **не совпадают** при $k \neq 1$ (z-компонента и нормировка другие). Но в вашем классе `Scaled...` метод `derivatives` вызывает `_compute_r_and_derivs`, который возвращает масштабированные $r, r', r''$. Поэтому `normal()`, наследованный от родителя, вычисляет нормаль **правильно** для масштабированной поверхности.

Аналогично для II формы:
```python
L = -r / sqrt(1 + r'^2)   # для базовой
L_scaled = -k*r / sqrt(1 + k^2*r'^2)   # для масштабированной
```

Это тоже правильно, т.к. `second_fundamental_form` вызывает `_compute_r_and_derivs`.

Так что геометрия `ScaledPiecewisePolynomialRevolution` корректна.

## 4. Теоретическое ограничение: коридор существования решения

Даже при правильной ТСН и исправном градиенте обратная задача может не иметь решения, если ТСН выходит за пределы **коридора** — множества точек, достижимых лучами из касательных плоскостей `E2`.

Для гомотетичной поверхности с совпадающими донцами и коэффициентом $k$:
- На днище ($u = a$) радиус $E1$ равен $k \cdot r(a)$, а $E2$ — $r(a)$. Разница $\Delta r = (k-1)r(a)$.
- На цилиндре разница постоянна: $\Delta r = (k-1)R_{cyl}$.
- На крышке зеркально днищу.

Если угол намотки таков, что свободный участок нити длиной $\lambda$ при данном направлении не достигает $E1$, решения нет. Но при $k=1.4$ и типичных углах это маловероятно.

## 1. Почему |Φ₀| = 2.02 мм: потеря соответствия «ЛУ ↔ ТСН»

В `corridor_3` (прямая задача) для каждой точки линии укладки `r(s)` на `E2` строится точка ТСН `R(s)` на `E1` по правилу:

> Луч из `r(s)` в направлении проекции касательной на `T_r E2` пересекает `E1` в `R(s)`.

По построению `(R(s) − r(s)) ⊥ m(r(s))`, то есть `Φ(s) = 0` **точно**.

В обратной задаче вы загружаете **только** точки `R(s)` из CSV. Начальная точка выбирается так:


## 2. Интерполяция ТСН кубическим сплайном вносит систематическую ошибку

В прямой задаче ТСН — это **кусочно-линейная** кривая: точки лежат на прямолинейных лучах. Но в обратной задаче вы строите:

```python
traj = Trajectory.from_points(points_tsn, method='cubic')
```

Кубический сплайн **сглаживает** углы между лучами. Следовательно:
- `traj.R(z)` отклоняется от истинных лучей;
- `traj.R'(z)` — касательная к сплайну — не совпадает с направлением луча;
- В `compute_dr_dz` величина `dΦ/dz = ⟨R'(z), m⟩` вычисляется неверно.

Это создаёт **систематический дрейф** предиктора. Корректор Ньютона компенсирует ошибку в `Φ`, но не исправляет ошибку в направлении движения. Накапливается несоответствие между параметром `z` (длина дуги ТСН) и истинной геометрией.

## 3. Критическая ошибка в `second_fundamental_form` (перепутаны `L` и `N`)

Это теоретически самый важный пункт. В `PiecewisePolynomialRevolution`:

```python
def second_fundamental_form(self, u, v):
    r, rp, rpp = self._compute_r_and_derivs(u)
    denom = np.sqrt(1 + rp * rp)
    L = -r / denom          # ← НЕВЕРНО
    M = 0.0
    N = rpp / denom         # ← НЕВЕРНО
    return L, M, N
```

Для поверхности вращения `X(u,v) = (r(u)cos v, r(u)sin v, u)` правильные формулы:

$$
L = \frac{r''}{\sqrt{1+r'^2}} \quad (\text{кривизна меридиана}), \qquad
N = \frac{-r}{\sqrt{1+r'^2}} \quad (\text{кривизна параллели})
$$

В коде `L` и `N` **перепутаны**. Проверка на цилиндре (`r'' = 0`):
- Должно быть: `L = 0`, `N = -r`.
- В коде: `L = -r`, `N = 0`.

Это означает, что `compute_grad_Phi` и `normal_curvature` вычисляют **неверное направление градиента** и неверную кривизну. Корректор Ньютона, вместо того чтобы двигаться вдоль нормального сечения, получает искажённое направление. На участках с большой кривизной (днище/крышка) ошибка катастрофична.

**Примечание:** ваше «исправление знака» в `compute_grad_Phi` (`inverse_method_fixed.py`) компенсировало часть этой ошибки для оригинальной `E1`, но при переходе к гомотетичной поверхности компенсация ломается, потому что масштабирование меняет соотношение `r''` и `r`.

## 4. Моё мнение: как использовать концепцию правильно

**Эллипсоид — это отличный тестовый стенд, но не промышленная замена `E1`.**

1. **Для отладки обратной задачи** — идеально. Генерируете `R(z)` на эллипсоиде, получаете идеально гладкую траекторию, и отлаживаете `compute_dr_dz`, `newton_corrector`, `DAEPredictor` без шума прямой задачи.

2. **Для верификации** — можно сравнить: решает ли обратная задача на эллипсоиде, а на кусочно-полиномиальной — нет. Если да, значит проблема была в шуме ТСН. Если нет — значит проблема в обратной задаче.

3. **Для промышленного применения** — поверхность безопасности обычно концентрическа оправке (или гомотетична). Эллипсоид — это математическая абстракция. Но после отладки на эллипсоиде можно вернуться к реалистичной `E1`, уже зная, что обратный алгоритм работает.

**Исправление** в `piecewise_polynomial_revolution_fixed_v2.py` (или родительском):
```python
def second_fundamental_form(self, u, v):
    r, rp, rpp = self._compute_r_and_derivs(u)
    denom = np.sqrt(1 + rp * rp)
    L = rpp / denom   # ← меридиан
    M = 0.0
    N = -r / denom    # ← параллель
    return L, M, N
```

И в `inverse_method_fixed.py` убедитесь, что `compute_grad_Phi` использует **чистую формулу** без лишних минусов:
```python
def compute_grad_Phi(surface, u, v, u_prime, v_prime, lam):
    L, M, N_ff = surface.second_fundamental_form(u, v)
    dPhidu = -lam * (L * u_prime + M * v_prime)
    dPhidv = -lam * (M * u_prime + N_ff * v_prime)
    return dPhidu, dPhidv
```

## Почему `u = −9.67` — классическая ошибка Python

В `inverse_method_fixed.py`, строка с клиппингом `u` скорее всего написана так:

```python
if u_min and u_max:          # ← БАГ!
    u_try = np.clip(u_try, u_min, u_max)
```

При `u_min = 0` (донце оправки) условие `if u_min:` даёт **`False`**, потому что `0` — falsy-значение в Python. Клиппинг **не срабатывает**, и корректор спокойно уходит в `u = −9.67`.

Правильно:
```python
if u_min is not None and u_max is not None:   # ← так
    u_try = np.clip(u_try, u_min, u_max)
```

## 1. `client_machine.py` — полный инженерный пайплайн «ЛУ → ТСН → кинематика станка»

Это **промышленный клиент**, который связывает геометрию намотки с кинематикой станка в одном скрипте.

**Этап 1. Геометрия и коридор**
- Загружает оправку `E2` и поверхность безопасности `E1` (кусочно-полиномиальные поверхности вращения).
- Загружает эталонную линию укладки `LU_data.mat`.
- Строит **коридор допустимых точек**: `CorridorMaxCalculator` (максимальное удаление от оси, ТСН на `E1`) и `CorridorMinCalculator` (минимальное удаление, `E2`).
- Фильтрует точки, где оба коридора валидны, получая согласованные пары «ЛУ на `E2` — ТСН на `E1`».

**Этап 2. Подготовка данных для станка**
- Вычисляет `theta_array` — азимут точек касания в глобальной системе (через `arctan2` + `unwrap`).
- Строит сплайн-интерполяцию ТСН и ЛУ как функции от натурального параметра `s`.

**Этап 3. Обратная задача кинематики**
- Создаёт объект `Machine3AxisExact_ODE` (модель 3-осного станка с кольцом).
- Решает **обратную кинематику**: по точке ТСН и радиусу оправки восстанавливает обобщённые координаты станка `(θ, Z, R, φ)`.
- Интегрирует систему ОДУ станка вдоль параметра `s`.
- Закомментирован блок уточнения итерационным методом (`Machine3AxisExact.inverse`).

**Этап 4. Визуализация и сохранение**
- **3D-сцена**: оправка, стена безопасности, ТСН, ЛУ, траектория центра кольца станка.
- **2D-графики** законов движения: `θ(s)`, `Z(s)`, `R(s)`, `φ(s)`.
- Сохраняет результаты в `kinematics_results.mat` и параметры станка в `machine_params.pkl`.

**Этап 5. Верификация**
- Решает **прямую задачу кинематики**: по найденным координатам станка вычисляет точки ТСН и сравнивает с исходными.
- Выводит среднюю и максимальную ошибку восстановления.

## 2. `client_cylinder_hybrid.py` — тестовый стенд на концентрических цилиндрах

Это **упрощённый демонстрационный клиент**, проверяющий обратную задачу на идеальной геометрии (цилиндры), без привязки к реальному станку.

**Геометрия**
- `outer = CylinderAnalytical(R_ext=3.0)` — поверхность безопасности `E1`.
- `inner = CylinderAnalytical(R_int=2.0)` — оправка `E2`.

**Траектория**
- Генерирует **винтовую линию** (`helical`) на внешнем цилиндре — это синтетическая ТСН.

**Обратная задача**
- Находит начальную точку на внутреннем цилиндре через `newton_corrector` (условие касательного луча).
- Настраивает **два предиктора**:

- Запускает `inverse_winding_hybrid` для восстановления ЛУ на внутреннем цилиндре.
- Выводит максимальную невязку `|Φ|`.

**Отличие от `client_machine.py`**
- Нет кинематики станка, нет коридоров, нет загрузки реальных данных.
- Чистая проверка алгоритма `inverse_winding_hybrid` на аналитически простой геометрии.

## 2. Информативные 2D-графики

### A. Отклонение ЛУ по нормали к E2
Самая важная метрика. Для каждой точки новой ЛУ `r_new(s)` находим ближайшую точку на исходной ЛУ `r_old(s)` и проецируем разницу на нормаль `m` к `E2`:

$$\Delta_n(s) = \big\langle r_{\text{new}}(s) - r_{\text{old}}(s),\; m_{\text{old}}(s) \big\rangle$$

**Интерпретация:** положительное — новая ЛУ выше (наружу) старой, отрицательное — ниже (внутрь). Должно быть `< 0.1` мм для приемлемой коррекции.

### B. Разница параметров на поверхности E2
$$\Delta u(s) = u_{\text{new}}(s) - u_{\text{old}}(s), \qquad \Delta v(s) = v_{\text{new}}(s) - v_{\text{old}}(s)$$

**Зачем:** показывает, в каких зонах (днище, цилиндр, крышка) коррекция вносит систематический сдвиг. На днище `Δu` критичен (там кривизна высокая), на цилиндре `Δv` — азимутальный сдвиг.

### C. Невязка связи для новой пары
$$\Phi_{\text{new}}(s) = \big\langle R_{\text{new}}(s) - r_{\text{new}}(s),\; m_{\text{new}}(s) \big\rangle$$

Должна быть такой же хорошей, как и для исходной (`~10^{-10}`). Если выросла — обратная задача для новой ТСН работает хуже.

### D. Длина свободного участка нити
$$\lambda(s) = \|R(s) - r(s)\|$$

Сравнение `λ_old(s)` vs `λ_new(s)`. Если коррекция сильно изменила `λ` — меняется натяжение/геометрия нити.

### E. Угол намотки α(s)
Угол между нитью и меридианом `E2`:
$$\cos\alpha = \frac{\langle \tau_{\text{thread}}, \tau_{\text{meridian}} \rangle}{\|\tau_{\text{thread}}\| \cdot \|\tau_{\text{meridian}}\|}$$

Сравнение `α_old(s)` vs `α_new(s)`. Для константного угла намотки (как в `ForwardWindingBuilder`) отклонения должны быть минимальны.

### F. Нормальная кривизна κ_n(s)
$$\kappa_n(s) = \frac{II}{I} = \frac{L\dot{u}^2 + 2M\dot{u}\dot{v} + N\dot{v}^2}{E\dot{u}^2 + 2F\dot{u}\dot{v} + G\dot{v}^2}$$

Сравнение кривизны исходной и новой ЛУ. Резкий скачок `κ_n` на новой ЛУ указывает на «излом» из-за ошибки интегрирования.

### G. Декартовые отклонения (ΔX, ΔY, ΔZ)
По отдельности — чтобы увидеть, в каком направлении преимущественно дрейфует ЛУ.

### H. Спектральный анализ (FFT)
FFT от `Δ_n(s)` или `Δu(s)`. Покажет, есть ли **резонансные частоты** в отклонении. Например, пик на частоте вращения оправки означает, что коррекция вносит систематическую ошибку на каждом обороте.

User: Что такое евклидово отклонение?
Kimi: **Евклидово отклонение** — это обычное прямолинейное расстояние в 3D между двумя точками.

В формулах клиента:
```python
dist_eucl = np.linalg.norm(lu_new - lu_orig, axis=1)
```

Это $\|\Delta\mathbf{r}\| = \sqrt{(\Delta x)^2 + (\Delta y)^2 + (\Delta z)^2}$ — расстояние от точки новой ЛУ до соответствующей точки исходной ЛУ.

**Чем отличается от $\Delta_n$ (отклонения по нормали):**

- **Евклидово отклонение** — полное 3D-расстояние. Содержит и нормальную составляющую (наружу/внутрь), и тангенциальную (сдвиг вдоль поверхности).
- **$\Delta_n$** — только проекция на нормаль к поверхности. Показывает, «вылезла» ли новая ЛУ из оправки или «провалилась» внутрь.

**Пример:** если новая ЛУ сдвинулась на 1 мм вдоль поверхности (по азимуту), но не оторвалась от неё:
- Евклидово отклонение = 1 мм.
- $\Delta_n$ = 0 мм.

Поэтому на графике евклидово отклонение всегда **≥** $|\Delta_n|$. Если они близки — значит, основная ошибка в нормальном направлении (дрейф от поверхности). Если евклидово отклонение сильно больше $|\Delta_n|$ — значит, есть ещё и тангенциальный сдвиг (смещение по поверхности).

## 2. Главное различие — параметризация

**Савин** интегрирует в **натуральном параметре $s$** (длина дуги линии укладки):
- Система (3.16): $\dfrac{d\mathbf{r}}{dz}$ и $\dfrac{ds}{dz}$ — ОДУ в 3D.
- Коррекция — сдвиг $\Delta s$ по длине дуги ЛУ.

**Мы** интегрируем в **параметре $z$** (длина дуги ТСН) и работаем в **координатах поверхности $(u,v)$**:
- DAE-система (4): дифференциальная часть для $(u,v)$ + алгебраическая связь $\Phi=0$.
- Коррекция — сдвиг $(\Delta u, \Delta v)$ в параметрическом пространстве.

## 3. Теоретический инсайт: эквивалентность при $\delta \to 0$

## 1. Почему взрывает с обоих концов

Теоретически возможны три сценария:

**А. Асимптотическое направление ($\kappa_n \to 0$)**
В точке $z^*$ градиент связи обращается в ноль:
$$\|\nabla\Phi\|^2 = g^{\alpha\beta}\,\Phi_{,\alpha}\,\Phi_{,\beta} \to 0$$
Корректор Ньютона теряет сходимость, знаменатель в $ds/dz$ (формула 3.41 Савина) обращается в ноль. Это **фундаментальная сингулярность**, общая для обоих направлений интегрирования.

**Б. Выход за границу коридора**
ТСН в точке $z^*$ проходит в зоне, недостижимой для лучей из касательных плоскостей $E2$. Геометрически: $\mathbf{R}(z^*)$ лежит вне огибающей семейства лучей. С обратной стороны — то же самое.

**В. Скачок на другую ветвь**
В точке $z^*$ существует **два** (или более) решения уравнения контакта $\Phi=0$. Интегрирование слева и справа сходится к **разным ветвям**, и в $z^*$ происходит разрыв. Это аналог **ударной волны** в гиперболических системах.

## 2. Стратегии заполнения пробела $(z_1, z_2)$

### Стратегия A: Рекурсивная бисекция (локальная)

**Идея.** Не пытаться пройти весь интервал сразу, а дробить его до тех пор, пока корректор не сойдётся.

**Алгоритм:**
```
function bridge(z_left, z_right, u_left, v_left, u_right, v_right):
    if |z_right - z_left| < dz_min:
        return interpolate(u_left, v_left, u_right, v_right)
    
    z_mid = (z_left + z_right) / 2
    R_mid = traj.R(z_mid)
    
    # Пробуем от левого конца
    u_mid, v_mid, Phi, conv = newton_corrector(E2, traj, u_left, v_left, z_mid)
    if conv and |Phi| < eps:
        # Пробуем от правого конца
        u_mid_r, v_mid_r, Phi_r, conv_r = newton_corrector(E2, traj, u_right, v_right, z_mid)
        if conv_r and |Phi_r| < eps:
            # Оба дошли — проверяем совпадение
            if ||r(u_mid,v_mid) - r(u_mid_r,v_mid_r)|| < tol:
                return bridge(z_left, z_mid, ...) + bridge(z_mid, z_right, ...)
    
    # Если не сошлось — дробим дальше
    return bridge(z_left, z_mid, ...) + bridge(z_mid, z_right, ...)
```

**Когда работает:** при сингулярности типа "острый пик", где корректор не сходится только в узкой зоне. При фундаментальном отсутствии решения — не работает.

---

### Стратегия B: Глобальная коллокация (на интервале)

**Идея.** Отказаться от пошагового интегрирования на $(z_1, z_2)$. Решить краевую задачу:

$$\text{Найти } \mathbf{r}(z) \in E2, \quad z \in [z_1, z_2]$$
$$\text{такую, что } \Phi(\mathbf{r}(z), \mathbf{R}(z)) = 0 \quad \forall z$$

**Постановка.** Параметризуем ЛУ на интервале полиномом или сплайном:
$$\mathbf{r}(z) \approx \sum_{i=0}^{N} \mathbf{c}_i \, B_i(z)$$
где $B_i$ — базисные функции (например, B-сплайны), $\mathbf{c}_i$ — узловые точки на $E2$.

**Условия коллокации** в точках $z_j \in [z_1, z_2]$:
$$\langle \mathbf{R}(z_j) - \mathbf{r}(z_j), \mathbf{m}(\mathbf{r}(z_j)) \rangle = 0, \quad j = 0, \dots, M$$

Это система $M+1$ нелинейных уравнений на $N+1$ векторов $\mathbf{c}_i$. Решается методом Ньютона-Рафсона в пространстве большой размерности.

**Преимущество:** не страдает от накопления ошибки интегрирования — всё решается одновременно.

**Недостаток:** требует хорошего начального приближения (берём интерполяцию между $\mathbf{r}(z_1)$ и $\mathbf{r}(z_2)$).

---

### Стратегия C: Оптический "мост" (гибридный)

**Идея.** В проблемной зоне отключаем DAE и используем **прямую трассировку лучей**.

**Алгоритм:**
1. В точке $\mathbf{r}_1 = \mathbf{r}(z_1)$ на $E2$ берём касательную $\tau_1$.
2. Проецируем $\tau_1$ на касательную плоскость в $\mathbf{r}_1$ — получаем направление луча.
3. Ищем пересечение луча с $E1$ — точка $\mathbf{R}_1^*$.
4. Если $\mathbf{R}_1^*$ близка к $\mathbf{R}(z_2)$, строим интерполяцию между $\mathbf{R}_1^*$ и $\mathbf{R}(z_2)$.
5. Иначе — двигаемся вдоль $E2$ от $\mathbf{r}_1$ с малым шагом, выпуская лучи, пока не найдём точку, луч которой попадает в окрестность $\mathbf{R}(z_2)$.

**Когда работает:** когда проблема в DAE-предикторе (он даёт скачок), а не в геометрии. Оптика обходит сингулярность, решая геометрическую задачу напрямую.

---

### Стратегия D: Регуляризация сингулярности

**Идея.** Если $\kappa_n \to 0$ — единственная проблема — ввести регуляризацию в знаменатель.

**Модификация формулы (3.41):**
$$\frac{ds}{dz} = \frac{\langle \mathbf{R}'_z, \mathbf{m} \rangle}{\|\mathbf{R}-\mathbf{r}\| \cdot (II + \varepsilon)} \alpha + \frac{\delta}{(II + \varepsilon)} \alpha k$$

где $\varepsilon > 0$ — малый параметр (например, $10^{-6}$). Это "сглаживает" асимптотическое направление, позволяя интегрированию продолжаться.

**Физический смысл:** при $\kappa_n \approx 0$ нить идёт почти по геодезической. Регуляризация заменяет точное геодезическое направление на приближённое.

---

### Стратегия E: Переключение на геодезическую модель

**Идея.** В зоне $\kappa_n \approx 0$ перейти от обратной задачи к **прямой** — построить геодезическую на $E2$ с сохранением угла намотки.

**Алгоритм:**
1. В точке $z_1$ фиксируем угол $\alpha_1$ между нитью и меридианом.
2. Интегрируем уравнение геодезики на $E2$ с углом $\alpha_1$ до точки $z_2$.
3. Проверяем, совпадает ли полученная ТСН с заданной $\mathbf{R}(z_2)$.
4. Если нет — корректируем $\alpha_1$ (метод стрельбы).

**Преимущество:** геодезическое уравнение не имеет сингулярности $\kappa_n = 0$ (асимптотические направления — это характеристики геодезических).
